# ECC Hook Fix — 2026-04-21

## Resumen

Claude Code CLI v2.1.116 en Windows fallaba todas las invocaciones de hooks de
la herramienta Bash con:

```
PreToolUse:Bash hook error
Failed with non-blocking status code:
C:\Program Files\Git\bin\bash.exe: C:\Program Files\Git\bin\bash.exe:
cannot execute binary file

PostToolUse:Bash hook error  (同上)
```

Resultado: `observations.jsonl` dejó de actualizarse después de
`2026-04-20T23:03:38Z` (la última entrada fue un `parse_error` de un problema
anterior de BOM en stdin).

## Causa raíz

`C:\Users\sugig\.claude\settings.local.json` tenía dos defectos:

### Defecto 1 — BOM UTF-8 + finales de línea CRLF

El archivo empezaba con `EF BB BF` (BOM UTF-8) y usaba terminadores de línea
`CRLF`. Este es el comportamiento por defecto de PowerShell `ConvertTo-Json |
Out-File`, y es lo que deja atrás `patch_settings_cl_v2_simple.ps1` cuando
reescribe el archivo.

```
00000000: efbb bf7b 0d0a 2020 2020 2268 6f6f 6b73  ...{..    "hooks
```

### Defecto 2 — invocación de bash.exe envuelta dos veces

La cadena de comando re-invocaba explícitamente bash.exe:

```json
"command": "\"C:\\Program Files\\Git\\bin\\bash.exe\" \"C:\\Users\\sugig\\.claude\\skills\\continuous-learning\\hooks\\observe-wrapper.sh\""
```

Cuando Claude Code lanza esto en Windows, la separación de argumentos no
preserva correctamente el token entre comillas `"C:\Program Files\..."`. El
espacio incrustado en `Program Files` parte `argv[0]`, y bash.exe termina
pasándose a sí mismo como archivo de script, produciendo:

```
bash.exe: bash.exe: cannot execute binary file
```

### Forma que funcionaba antes (para referencia)

Antes de ejecutar `patch_settings_cl_v2_simple.ps1`, el comando era simplemente:

```json
"command": "C:\\Users\\sugig\\.claude\\skills\\continuous-learning\\hooks\\observe.sh"
```

Claude Code en Windows detecta `.sh` y lo invoca mediante Git Bash por sí
mismo — no hace falta envolverlo manualmente con `bash.exe`.

## Fix

`C:\Users\sugig\.claude\settings.local.json` se reescribió como UTF-8 (sin
BOM), con finales de línea LF, y con el comando apuntando directamente al `.sh`
wrapper y pasando la fase del hook como argumento simple:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "C:/Users/sugig/.claude/skills/continuous-learning/hooks/observe-wrapper.sh pre"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "C:/Users/sugig/.claude/skills/continuous-learning/hooks/observe-wrapper.sh post"
          }
        ]
      }
    ]
  }
}
```

Beneficio secundario: el argumento `pre` / `post` ahora se enruta a la variable
`HOOK_PHASE` de `observe.sh`, así que los eventos se registran correctamente
como `tool_start` vs `tool_complete` (antes todo se registraba como
`tool_complete`).

## Verificación

Invocación directa del nuevo formato de comando, emulando ambas fases del hook:

```bash
# Ruta PostToolUse
echo '{"tool_name":"Bash","tool_input":{"command":"pwd"},"session_id":"post-fix-verify-001","cwd":"...","hook_event_name":"PostToolUse"}' \
  | "C:/Users/sugig/.claude/skills/continuous-learning/hooks/observe-wrapper.sh" post
# exit=0

# Ruta PreToolUse
echo '{"tool_name":"Bash","tool_input":{"command":"ls"},"session_id":"post-fix-verify-pre-001","cwd":"...","hook_event_name":"PreToolUse"}' \
  | "C:/Users/sugig/.claude/skills/continuous-learning/hooks/observe-wrapper.sh" pre
# exit=0
```

`observations.jsonl` ganó:

```
{"timestamp":"2026-04-21T05:57:54Z","event":"tool_complete","tool":"Bash","session":"post-fix-verify-001",...}
{"timestamp":"2026-04-21T05:57:55Z","event":"tool_start","tool":"Bash","session":"post-fix-verify-pre-001","input":"{\"command\":\"ls\"}",...}
```

Ambas fases ahora producen eventos tipados correctamente.

**Nota sobre verificación en vivo del CLI:** los cambios de settings surten
efecto en el siguiente lanzamiento de sesión del CLI `claude`. Reinicia el CLI y
haz una llamada a la herramienta Bash para confirmar que aparecen nuevas filas
en `observations.jsonl` desde la sesión real del CLI.
